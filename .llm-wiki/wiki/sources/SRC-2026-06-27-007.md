# Logging and audit (docs/en/07)
> _Original: `docs/en/07-logging-audit.md` (~8 KB)_

## Summary

Observability in sign-verify-2 rests on three pillars: **structured application logs** (JSON to STDOUT, logback + MDC, §6.1), **metrics** (Actuator/Prometheus, §6.4), and a persistent **audit log** queryable via API (§6.3). Errors are emitted as `application/problem+json` (RFC 9457, §6.2). 

> ⚠️ **Current implementation status (important):** the `audit_log` table, the `AuditService` write component, and the read API exist and are indexed, but `AuditService` is **not yet wired** into operational paths (verification, key management, TSL refresh) — so the table may currently be empty. Operational traceability is presently provided by structured logs with MDC. The audit infrastructure is ready to be hooked in.

## Key Takeaways

- **Structured logs (§6.1):** JSON to STDOUT, per-request context via **MDC** (correlation/principal), rotation configured.
- **Error handling (§6.2):** `application/problem+json` (RFC 9457) with stable `type` URNs.
- **Audit log (§6.3):** `audit_log` table (`AuditLog`): `id`, `occurredAt`, `principalType`/`principalId` (or `SYSTEM`), `action`, `targetType`/`targetId`, `success`, `details` (JSON), `ipAddress`. Query via `GET /api/v1/audit-log` (**PRIVILEGED**), filters: `principalId`, `action`, `from`/`to`, `targetType`/`targetId`, `success`, paging; sorted by `occurredAt` desc.
- **Metrics (§6.4):** Micrometer → `/actuator/prometheus`; custom metrics for verification outcomes, concurrency, etc.
- **Status caveat:** audit writes not yet wired into operations — see the `wire-audit-log` plan.
