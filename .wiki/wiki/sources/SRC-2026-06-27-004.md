# Trusted Certificates API — TSL (docs/en/04)
> _Original: `docs/en/04-trusted-certificates.md` (~8 KB)_

## Summary

Operational guide to **Trusted List (TSL/LOTL)** management and the queryable trusted-certificate mirror in sign-verify-2. Explains the refresh model (§3.1), the TSL status endpoint (§3.2), forcing a refresh (§3.3), listing/viewing trusted certificates (§3.4–3.5), the permissions model (§3.6), and the **OJ keystore** that anchors LOTL trust (§3.7), including how to rebuild it and how to diagnose national TSLs failing with `PKIX path building failed`.

The trusted-certificate table is a **queryable mirror** populated after a TSL refresh (see design doc §7, [[sources/SRC-2026-06-27-002]]); refreshes are scheduled, ShedLock-coordinated, and atomic (hot swap). The OJ (Official Journal) keystore holds the certificates DSS uses to verify the LOTL signature.

## Key Takeaways

- **Refresh model (§3.1):** scheduled (cron) + on-startup + manual force; ShedLock ensures single execution across instances; results written atomically to the mirror, with `tsl_refresh` history rows (`SUCCESS`/`PARTIAL`/`FAILED`, counts of certs added/removed/unchanged).
- **TSL status (§3.2):** reports last refresh time, per-source outcome, and certificate counts.
- **List/detail (§3.4–3.5):** query trusted certificates (subject, country, qualifiers, service status, `last_seen_at`).
- **Permissions (§3.6):** read of trusted certs/status is available to authenticated principals; force-refresh and keystore management are **PRIVILEGED**.
- **OJ keystore (§3.7):** `oj-keystore.p12` is the LOTL trust anchor; must be rebuilt when a new Official Journal is published. National TSLs failing with `PKIX path building failed` usually indicate a stale/missing OJ keystore or a pivoted LOTL URL.
