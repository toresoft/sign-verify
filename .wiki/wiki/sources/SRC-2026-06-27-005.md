# Authentication and authorization (docs/en/03)
> _Original: `docs/en/03-authentication.md` (~8 KB)_

## Summary

How callers authenticate to sign-verify-2. The service is **stateless** (no sessions, CSRF disabled) and supports **two mechanisms in one filter chain**: **API key** (`X-API-Key: sv_<prefix>_<body>`, always active) and optional **OAuth2 JWT** (`Authorization: Bearer <jwt>`, toggled by `app.security.oauth.enabled`). A unified Principal abstraction (`API_KEY` / `OAUTH_USER`) feeds role checks. Covers API key format & bootstrap (§2.2), the "last privileged key" invariant, key management API, and OAuth configuration/usage (§2.3).

The **API-key filter runs before** the OAuth chain: if a valid `X-API-Key` is present the principal is `API_KEY`; otherwise OAuth (if enabled + valid Bearer) yields `OAUTH_USER`. Both present → API key prevails (logged as warning). Missing credentials → `401 auth.missing-credentials`.

## Key Takeaways

- **API key format:** `sv_<prefix>_<body>`; the `<prefix>` is indexed + unique for fast lookup; the full key is bcrypt-hashed at rest and returned in plaintext **only once** at creation.
- **Bootstrap key:** a PRIVILEGED bootstrap key is generated on first startup (if none exists) and written to `app.security.bootstrap-key-file` (mode 0600).
- **"Last privileged key" invariant:** the last enabled PRIVILEGED key cannot be removed/disabled, guaranteeing administrative access is never locked out.
- **OAuth2 (§2.3):** Resource Server with `issuer-uri`, JWKS validation; the Principal is mapped from JWT claims (`app.security.oauth.role-claim`, `app.security.oauth.privileged-values`).
- **Roles:** `PRIVILEGED` / standard; `@EnableMethodSecurity` guards privileged endpoints; behavior is identical for both principal types.
